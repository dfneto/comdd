\chapter{Introdução}
\label{introducao}

\lettrine[lines=3]{O}\ Desenvolvimento Orientado a Modelos (MDD - \textit{Model-Driven Development}) é uma combinação de programação generativa, linguagens específicas de domínio e transformações de software, conceitos que já vem sendo explorados há algum tempo  \cite{Neighbors:thesis:1980,Lucredio:2006:wdbc}. Seu objetivo é reduzir a distância semântica que existe entre o domínio do problema e o domínio da implementação/solução, utilizando modelos mais abstratos que protegem os desenvolvedores de software das complexidades inerentes à plataforma de implementação \cite{France:2007:fose}.

Nessa abordagem, os artefatos com modelos são utilizados para expressar conceitos de forma mais eficiente, pois inclui  transformações que são utilizadas para gerar automaticamente os respectivos artefatos de implementação os quais refletem a solução expressa nos modelos \cite{Schmidt:2006:ieeecomputer}. Um dos principais benefícios desta abordagem é a reutilização. Mo\-de\-los condensam conhecimento produzido durante atividades de análise, projeto e implementação, de uma forma (idealmente) independente da plataforma de implementação. Assim, reutilizando-se um modelo, este conhecimento pode ser mais facilmente reaproveitado em outros contextos \cite{Sherif:2006:ijim}.

A reutilização (ou reúso) de software é uma atividade altamente cola\-borativa em sua essência, pois normalmente o reutilizador de um artefato não é a mesma pessoa (e possivelmente não precisa pertencer à mesma equipe) que o produtor do artefato \cite{Almeida:caise:2005}. Mesmo a produção de artefatos reutilizáveis envolve a colaboração de diversos profissionais, tais como o analista de domínio, projetista de domínio, implementadores, especialistas do domínio, além dos diversos \textit{stakeholders} envolvidos no processo.

Neste contexto, considerando-se os aspectos cola\-bo\-rativos da reutilização de software em uma abordagem orientada a modelos, existem diferentes linhas de pesquisa que merecem uma investigação mais detalhada, dentre as quais podemos destacar:

%\color{red}
\begin{enumerate}
\item Tecnológico: investigação de ferramentas de modelagem e transformação de software que executem em um ambiente colaborativo, assim como as ferramentas de suporte, tais como controle de versões, alterações e evolução no contexto do desenvolvimento orientado a modelos;
\item Processual: investigação e definição de atividades e tarefas colaborativas para o desenvolvimento orientado a modelos, visando a reutilização;
\item Gerencial: investigação dos aspectos relativos à gerência de um processo colaborativo para o desenvolvimento orientado a modelos. Inclui a definição de métricas e métodos para planejamento/controle do processo;
\item Qualitativo: questões sobre os procedimentos e tecnologias de acesso e  comunicação, bem como a adequação às normas e legislação vigentes devem ser assegurados para os diversos tipos de envolvidos na colaboração.
\end{enumerate}

Outra tendência promissora de utilização de MDD é em sistemas embarcados \cite{Liggesmeyer2009}. Gerenciar o rápido crescimento da complexidade do desenvolvimento de sistemas embarcados é uma das tarefas mais desafiadoras para aumentar a qualidade do produto, diminuir o tempo até a chegada do produto ao mercado e reduzir os custos de desenvolvimento. O MDD se apresenta como uma das abordagens que têm emergido nas últimas décadas e encontrada grande aceitação na comunidade de engenharia de software.

Um software embarcado raramente é desenvolvido como um produto singular, mas como um elemento de um sistema embarcado, que é composto de outros elementos (circuitos elétricos, eletrônicos, peças mecânicas, etc). Assim, o desenvolvimento de sistemas embarcados torna-se mais colaborativo que o desenvolvimento de sistemas de informação, em geral, pois para o desenvolvimento de um sistema embarcado é necessário não somente os desenvolvedores do software, como também a intervenção dos demais pro\-fis\-si\-o\-na\-is envolvidos, como: os engenheiros elétricos, eletrônicos, mecânicos, da computação, dentre outros\cite{Liggesmeyer2009}.

Ainda no contexto de suporte a trabalhos colaborativos, percebe-se que atualmente, tem havido um movimento de migração de sistemas de desktop para aplicações web. Observando o histórico desses sistemas, acredita-se que num futuro não muito distante, ter-se-á o desenvolvimento de softwares colaborativos via web com muito mais vigor \cite{Bozzon}.

%Assim, o MDD tem se mostrado uma abordagem madura e que traz benefícios para o desenvolvimento de sistemas. Seu foco nos modelos eleva o nível de abstração do desenvolvimento possibilitando aumento de produtividade, comunicação, interoperabilidade, entre outros. Embora o MDD esteja sendo empregado com sucesso, ainda há uma carência nas ferramentas que auxiliem seu uso, principalmente relativas a colaboração em MDD.

A colaboração é uma atividade que pode trazer benefícios para o MDD e conseqüentemente para a equipe e projeto, como se deseja comprovar neste trabalho. Ela pode ser imprescindível em sistemas extensos e complexos. Portanto,  \textbf{CoMDD - \emph{Colaborative Model-Driven Development}} é proposto como um ambiente de desenvolvimento colaborativo orientado a modelos. Espera-se também, com este ambiente, levantar quais são esses benefícios, com foco principal nos domínios de sistemas embarcados (SE) e no domínio de tecnologia de informação (TI), em especial o domínio de modelos Entidade-relacionamento (MER).  

Contudo, para desenvolver este ambiente, é preciso atentar para os problemas que a colaboração implica. Assim, problemas relativos ao versionamento, controles de permissão e autorizações de acesso, \textit{design} das decisões, \textit{merge}, comparação entre modelos, modelagem e transformação distribuída, foram identificados e pretende-se, no decorrer do projeto buscar por soluções para  tais itens.  

Observando os trabalhos nessa linha de pesquisa e analisando os benefícios em potencial que esta abordagem pode trazer, espera-se que este trabalho possa contribuir para o processo de desenvolvimento de softwares, para a equipe de desenvolvimento e para a Engenharia de Software.
