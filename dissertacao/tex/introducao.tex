\section{Contextualização do Trabalho}
\label{problema}

%O desenvolvimento orientados a modelos é ... 
O desenvolvimento orientado a modelos (\textit{Model Driven Development} - MDD) é um método de desenvolvimento de software que foca nos modelos, como os principais artefatos de desenvolvimento, e nas transformações desses modelos para gerar outros modelos em diferentes níveis de abstração, até mesmo código-fonte. O objetivo do MDD é reduzir a distância semântica existente entre o domínio do problema e o domínio da implementação/solução, utilizando modelos mais abstratos que protegem os desenvolvedores de software das complexidades inerentes às plataforma de implementação \cite{Teppola2009, France2007}. %tepoola:ok, france:NOTok

% e é bom por causa disso ...
A principal vantagem do MDD é poder expressar modelos usando conceitos menos vinculados a detalhes de implementação, além do fato de modelos serem mais próximos do domínio do problema, o que os torna mais fáceis de se especificar, entender e manter. Em alguns casos, ainda é possível os especialistas do domínio produzirem os sistemas ao invés dos especialistas da tecnologia de implementação \cite{Selic2003, Sriplakich2006}

%--------------------> Ver outros autores que citam os benefícios do MDD
%e disso ...
O MDD traz benefícios como: produtividade, portabilidade, manutenção etc \cite{Kleppe2003,Lucredio2009, Bendix2009}. Há diversos trabalhos \cite{Aho2009,France2007,Kleppe2003,Lucredio2009,Mellor2003,Sanchez2009,Stahl2006, Liggesmeyer2009} que propõem e que usam o MDD como abordagem para desenvolvimento de software. Entre as áreas da indústria que usam MDD, a área de sistemas embarcados já se beneficia desta abordagem antes mesmo do surgimento da UML ou da \textit{Model Driven Architecture} - MDA\footnote{A MDA (Model Driven Architecture) é um pradão da OMG como abordagem do MDA, com uso da UML para a modelagem} \cite{Liggesmeyer2009}.

Entretanto, em relação a modelos textuais\footnote{Lembrando que modelos podem ser textuais ou gráficos.}, como equipes de desenvolvimento de software trabalham? 

%Apesar do trabalho de Bendix e Emanuelson \cite{Bendix2009} dizer que há uma carência de ambientes que auxiliem o desenvolvimento orientado a modelos de maneira colaborativa -> que as ferramentas para MDD são quase inexistentes <-; quando trabalha-se com modelos textuais as ferramentas para MDD são as mesmas que existem para as abordagens tradicionais como por exemplo Java, C, C++ e etc. Ou seja, assim como pode-se usar o Eclipse\footnote{http://eclipse.org/} como IDE de desenvolvimento e o SVN\footnote{http://subversion.tigris.org/} para compartilhamento de código, pode-se usar ambas ferramentas para desenvolver usando MDD. Os modelos textuais são textos da mesma forma que um código-fonte Java é texto, assim não há razão técnica para não usar-se o uma IDE que suporte MDD e um sistema de controle de versão\footnote{Este trabalho usará os termos sistema de controle de versão ou SCV ou sistema de versionamento como sinônimos.} qualquer.

Segundo Bendix e Emanuelsson (2009), as soluções existentes ainda são incipientes em termos de ferramentas e processos necessários para auxiliar os desenvolvedores de MDD em seus trabalhos \cite{Bendix2009}. Entretanto, uma possibilidade para se trabalhar com MDD em uma equipe de desenvolvimento é usar uma IDE, para o desenvolvimento de modelos e geração de código-fonte, e um sistema de versionamento\footnote{Este trabalho usará os termos sistema de controle de versão ou sistema de versionamento como sinônimos.}, para compartilhar modelos entre os membros da equipe. 
%Esta abordagem, a de uma IDE associada à um sistema de versionamento é comum na indústria.

%Contudo, (aqui vem todos os problemas): Problemas dos svns da vida (merge por ex)
Contudo, os sistemas de versionamento podem causar conflitos quando dois ou mais desenvolvedores modificam o mesmo documento e submetem ao servidor\footnote{Isso no caso de uma política de versionamento \textit{otimistic}, a qual permite o desenvolvimento concorrente \cite{Sarma2003}.}. 

Neste momento, desenvolvedores têm que dedicar certo tempo para resolver os conflitos, e às vezes esses conflitos só podem ser resolvidos manualmente. Além disso, para os desenvolvedores saberem como resolver o conflito corretamente eles precisam estar em contato entre si e para decidirem como realizar o \textit{merge} os desenvolvedores precisam estar disponíveis no mesmo momento \cite{Prudencio2012}. 

Observa-se que quase todas as ferramentas de desenvolvimentos são projetadas para desenvolvedores e poucas delas servem para desenvolvedores periféricos participarem. Como resultado, há várias barreiras para os desenvolvedores periféricos quando eles desejam contribuir para um projeto \cite{Xiao2007}. Os próprios stakeholders ficam limitados em sua participação no desenvolvimento uma vez que pouco ou não compreendem código-fonte. Um exemplo dessa barreira é a necessidade de instalação de software locais.

%programas necessarios instalados,
Ao se usar IDEs e sistemas de versionamento, é preciso que cada desenvolvedor tenha instalado esses softwares em seus computadores. O que significa dizer que se um \textit{stakeholder} estiver interessado em acompanhar o desenvolvimento de um software ele também precisa instalar essas ferramentas\footnote{Em um dos estudos de caso deste trabalho, descrito no Seção \ref{estudo_de_caso_ii}, dois desenvolvedores experimentes levaram cerca de quarenta minutos para baixar, instalar e configurar o Eclipse e o plugin do SVN}. Embora isso seja comum para desenvolvedores, esta tarefa se torna difícil e trabalhosa para não desenvolvedores.

Além da instalação e configuração, o \textit{stakeholder} deve saber como funcionam essas ferramentas. Se ele quiser participar da edição dos modelos\footnote{Este trabalho parte do uso de MDD como abordagem de desenvolvimento e por isso está-se referindo ao MDD.} deverá ainda conhecer como funciona a geração de códigos da IDE, como fazer \textit{check-in}\footnote{Também chamado de \textit{commit}, segundo Sussman et al. \cite{BenCollins-Sussman2011} é a operação de enviar mudanças da cópia local para a cópia no repositório.}, \textit{check-out}\footnote{Segundo Sussman et al. \cite{BenCollins-Sussman2011} é a operação de fazer uma cópia local da cópia do repositório.}, entre outros termos e suas variações de nome\footnote{Por exemplo, no plugin do SVN para o Eclipse, o \textit{check-out} é quando faz-se uma cópia completa de todos os arquivos do servidor e \textit{update} quando apenas atualiza as modificações.}. Assim, embora para um usuário não desenvolvedor, trabalhar com modelos seja mais fácil do que com código-fonte \cite{Selic2003}, essa vantagem é minimizada quando há a necessidade do uso dessas ferramentas.

%processo de check-in, check-out, instalação muito demorado
Além disso, todo esse processo de instalação, de \textit{check-in}/\textit{check-out} etc, é relativamente demorado, passando a configurar como mais uma desvantagem somada a necessidade de conhecimento desses conceitos. Jones e Scaffidi (2011) apóiam esta ideia quando afirmam que cientistas raramente usam controle de versão devido ao alto custo inicial para configuração desses sistemas. %talvez seja melhor colocar esse referencia em outro lugar.

%ainda sao ferramentas que requerem uma curva de aprendizado grande no inicio, principalmente porque são varios os tipos e cada um tem uma abordagem diferente de uso, por exemplo, o svn é diferente do git que é diferente do da microsoft, ainda tudo isso para alguem que nao é desenvolvedora e que trabalha com programação é complicado de acompanhar o projeto.
E mesmo que o usuário receba um treinamento, as abordagens são diferentes entre si. Por exemplo, a abordagem do SVN\footnote{http://subversion.tigris.org/} é diferente da abordagem do Git\footnote{http://git-scm.com/}, que é diferente da abordagem usada no \textit{Team Foundation Service} da Microsoft\footnote{http://tfspreview.com/}, de modo que além da necessidade do conhecimento da instalação, configuração, conceitos e uso, é também necessário o conhecimento da lógica de funcionamento dessas abordagens e de suas ferramentas. Todas essas informações necessárias tornam o processo de versionamento um processo de curva de aprendizado inicial alta.

Um outro problema encontrado no desenvolvimento de software está relacionado a manter a documentação atualizada. Geralmente, o que ocorre é que a documentação usada para desenvolver o software fica desatualizada em relação ao código-fonte \cite{Lucredio2009, Xiao2007}. É um desafio para equipes de desenvolvimento manter o código-fonte e documentação sincronizados ao longo de todo o ciclo de vida do desenvolvimento, mas que compensa \cite{Xiao2007}.

Uma documentação bem elaborada beneficia a todos os desenvolvedores do projeto, mas manter uma boa documentação pode ser uma tarefa difícil, custosa e maçante. Todos os artefatos envolvidos na documentação requerem uma contínua e constante revisão e modificação ao longo do ciclo de vida do software com o intuito de preservar seu valor e consistência \cite{Aguiar2005}.

%wiki como solução: os recursos como histórico de versões, comentários e compartilhamento de código podem ser alcançados através de uma wiki
Por outro lado, as wikis são ferramentas conhecidas por serem simples de usar e ágeis para colaborar \cite{Abeti2009,Tetard2009}. Dependendo do problema, o uso de uma wiki ao invés de uma IDE associada a um SVN pode contornar todos os problemas descritos anteriormente, pois as wikis funcionam em um navegador, possuem um controle de versões com histórico simplificado, possibilitam inserção de comentários e permitem travar a edição concorrente, o que evita conflitos e \textit{merge}. Assim, implementar na wiki um suporte a modelagem e geração de código-fonte, para que as wikis possam servir de plataforma de MDD colaborativo, se mostra um objetivo promissor.

%tendencia de tudo na nuvem
Ainda, analisando o histórico de aplicações como músicas, vídeos, jogos, o armazenamento de arquivos, entre outros, observa-se uma tendência das aplicações migrarem para Web, como exemplo disso tem-se o Grooveshark®\footnote{grooveshark.com} (reprodutor de músicas on-line), o Youtube® \footnote{youtube.com} (reprodutor de vídeos on-line), o Dropbox®\footnote{dropbox.com} (armazenador de arquivos on-line) e até mesmo IDEs estão convergindo para a web, como é o caso do Cloud9\footnote{http://c9.io/}, por exemplo. Assim, ter uma ferramenta possível de editar, compartilhar e versionar modelos, gerar código-fonte e ainda permitir a entrada de comentários, pode beneficiar desenvolvedores e \textit{stakeholders}.

Contudo, seria possível uma wiki ser usada como uma ferramenta de desenvolvimento e compartilhamento de modelos, de forma que seja capaz de atender às necessidades dos usuários das IDEs e dos sistemas de versionamento? A partir deste questionamento, este trabalho definiu uma abordagem colaborativa orientada a modelos (\textit{Collaborative Model Driven Development} - CoMDD), a qual usa uma wiki para apoiar o MDD de uma equipe de desenvolvimento nas tarefas de modelagem, geração de código, versionamento e comunicação.

\section{Hipótese e objetivo}
\label{hipotese}
%conclusão da introdução: assim, com o uso de mdd resolvemos vários problemas como ... e com o uso de wikis resolvemos outros problemas como ..., por isso o objetivo deste trabalho é ..
A partir do cenário das dificuldades de desenvolvimento colaborativo de software usando MDD, definiu-se a seguinte hipótese que norteia este trabalho:

É possível usar uma wiki, ao invés do processo tradicional de desenvolvimento\footnote{Definimos processo tradicional de desenvolvimento como sendo o uso de uma IDE e de um sistema de controle de versão \textit{otimistic}}, para desenvolver um sistema colaborativamente usando MDD\footnote{No caso estamos nos referindo a DSLs como abordagem de MDD, conceitos a serem descritos no Capítulo \ref{revisaoBibliografica}.}.

%hipótese 3: as pessoas usam a wiki para uma série de fatores, como discussoes, documentações. Principalmente em virtude da sua simplicidade e agilidade (de acordo com o dicionario, simplicidade é ..., agilidade é ...), mas será que elas poderiam serem usadas para produzir codigo fonte? Ou seja, sera que as wikis podem servir como uma ide e ainda assim manter suas caracteristicas/beneficios de agilidade/simplicidade?

O objetivo deste trabalho é evidenciar que a hipótese de pesquisa é verdadeira para um caso específico (a ser tratado nos estudos de caso) e com isso incentivar o uso do CoMDD por equipes de desenvolvimento.

\section{Escopo}
\label{}
Está fora do escopo deste trabalho comparar o desenvolvimento tradicional (que usa linguagens de amplo propósito, como Java, por exemplo) com o MDD, uma vez que a literatura já discute sobre isso e sobre os benefícios em termos de produtividade que o MDD agrega para o desenvolvimento de software \cite{Aho2009,France2007,Kleppe2003,Lucredio2009,Mellor2003,Sanchez2009,Stahl2006, Liggesmeyer2009}.

Este trabalho menciona sobre a comunicação entre os integrantes de uma equipe, desenvolvedores ou não, e entende que é importante promover e incentivar a comunicação; contudo, não faz parte do escopo deste trabalho desenvolver ferramentas ou um processo de comunicação entre os desenvolvedores e nem um estudo a fundo, mas apenas incentivar a partir da forma que o CoMDD é concebido e incentivar o uso de mensageiros instantâneos ou ferramentas de vídeo-conferência para promover a comunicação.
